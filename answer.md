## 1. 文档目标

本文档基于以下两类输入形成：

- 赛题要求：`problem.md`

> 交付一个“基于现有结果缓存的离线校园安防预警 Demo”，而不是从零实现完整实时多模型推理系统。

## 2. 结论先行

### 2.1 推荐方案

推荐采用：

**“结果复用型单路视频离线预警 Demo”**

即：

1. 不重跑 GEBD、LLaVA、DeepSeek、ImageBind 全链路。
2. 直接复用仓库中已存在的最终异常分数、推理文本、指标结果。
3. 在此基础上补一层：
   - 校园行为类别映射
   - 预警事件生成
   - 视频回放与异常曲线展示
   - 识别日志导出
   - 异常片段切片导出
   - 历史记录缓存

### 2.2 为什么这是 2 小时内最优解

因为当前仓库虽然已经具备完整论文链路，但全链路推理成本很高：

- `EfficientGEBD` 负责边界检测
- `LLaVA-NeXT` 负责视频片段描述
- `DeepSeek-R1` 负责异常推理
- `ImageBind` 负责相似度细化
- `correlation_eval.py` 负责最终融合

这条链路适合论文复现，不适合 2 小时内从头交付比赛 Demo。原因如下：

- 视频数据量很大，本地 `UCF_CRIME_TEST_VIDEO_DIR` 已接近 100G。
- `deepseek_batch_infer.py` 默认就是大模型推理路径，不适合作为 2 小时演示主路线。
- 题目要求覆盖多人交互、角色区分、实时预警、前端展示、日志导出等，若从算法层补齐，工作量远超 2 小时。

但当前仓库里已经有非常关键的现成资产：

- 本地视频：`UCF_CRIME_TEST_VIDEO_DIR/`
- 已计算好的中间结果和最终结果：`result/UCF_Crime_test/`、`result/MSAD_test/`
- 逐帧异常分数
- 节点级描述文本
- LLM 推理文本
- 视频级评估指标

因此，最省时且最稳妥的方案，是把当前仓库包装成一个**离线回放式智能预警系统**。

## 3.1 赛题关键要求

根据 `problem.md`，题目希望系统具备：

- 视频流接入与在线处理

- 历史记录缓存

## 3.2 2 小时版本必须压缩的范围

2 小时内不可能同时完成：

- 真正的多模型在线推理
- 多人跟踪
- 新行为类别训练
- 高精度校园专用分类器
- 真正的多路实时 1080P 处理

因此需要把目标收缩为：

### 2 小时可交付范围

- 单路视频回放式异常预警
- 使用现有结果缓存完成“准实时回放展示”

### 明确不做

- 对任意新视频做完整模型推理
- 真实在线流式分析
- 人员检测与跟踪重建
- 霸凌中的攻击方/受害方角色细分

## 5. 总体研发目标

## 5.1 最终交付物定义

交付一个最小可演示系统，名称建议为：

**CampusGuard-Demo**

系统具备以下能力：

1. 选择一个本地视频样本进行分析展示。
2. 读取该视频对应的已缓存异常分数。
3. 自动生成异常事件段。
4. 将异常事件映射到校园风险标签。
5. 展示事件说明、时间范围、风险等级、原因文本。
6. 导出日志文件。
7. 导出异常视频切片。
8. 保存历史分析记录。

## 5.2 研发目标边界

本方案是：

- 一个**演示型系统**
- 一个**离线回放式预警系统**
- 一个**已有模型结果的业务化包装层**

本方案不是：

- 一个完整在线推理平台
- 一个校园专用训练模型
- 一个真实部署版安防平台


## 6. 推荐技术路线

## 6.1 核心思路

整体采用“结果驱动”的 6 层结构：

```text
本地视频 / 结果缓存
    -> 结果加载层
    -> 事件生成层
    -> 校园标签映射层
    -> 预警展示层
    -> 导出与缓存层
```

## 6.2 方案结构图

```text
Video File

      |
      v

```


## 7. 功能设计

## 7.1 输入层

### 输入内容

- 本地视频文件
- 对应的视频异常分数 JSON
- 对应的描述文本 JSON
- 对应的推理文本 JSON

### 2 小时版本建议

输入只支持两类来源：

1. 预设样本视频
2. 已有结果缓存的视频

也就是说：

- 可以做“本地视频选择”
- 但不做“任意视频上传后现场跑完整推理”

这是当前时间约束下最合理的取舍。

## 7.2 结果加载层

### 功能目标

统一读取并组织以下数据：

- 视频名
- 视频路径
- 数据集来源
- 视频类别
- 逐帧异常分数
- clip 级推理文本
- clip 级描述文本
- 视频级评估指标

### 主要数据来源

- 最终异常分数：`ense_refine*.json`
- 推理文本：`DeepSeek.../maxf64*.json`
- 描述文本：`LLaVA.../maxf64*.json`
- 指标：`00vid_metric.json`

### 输出数据结构建议

```json
{
  "video_name": "Fighting018_x264.mp4",
  "video_path": "UCF_CRIME_TEST_VIDEO_DIR/Fighting018_x264.mp4",
  "dataset": "UCF_Crime",
  "source_class": "Fighting",
  "frame_scores": [0.01, 0.01, 0.03, 0.52, 0.88],
  "captions": {
    "0, 120": "..."
  },
  "reasoning": {
    "0, 120": [0.8, "..."]
  },
  "metrics": {
    "roc_auc": 0.97,
    "pr_auc": 0.99
  }
}
```

## 7.3 事件生成层

### 功能目标

把逐帧异常分数转换为可读的“预警事件”。

### 处理流程

1. 读取 `frame_scores`
2. 对分数做轻量平滑
3. 使用阈值切分异常区间
4. 合并相邻区间
5. 过滤过短区间
6. 生成事件对象

### 建议参数

- 平滑窗口：15 到 30 帧
- 预警阈值：0.65
- 高危阈值：0.80
- 最短事件时长：1.5 秒
- 合并间隔：1 秒以内自动合并

### 输出数据结构建议

```json
{
  "event_id": "evt_0001",
  "video_name": "Fighting018_x264.mp4",
  "start_frame": 320,
  "end_frame": 465,
  "start_sec": 10.67,
  "end_sec": 15.50,
  "peak_score": 0.91,
  "mean_score": 0.82
}
```

## 7.4 校园行为映射层

### 功能目标

把公开数据集中的类别转换为校园安全语义标签。

### 推荐映射表

| 数据集原类 | 校园映射类 | 风险级别 | 说明 |
| --- | --- | --- | --- |
| Fighting | 打架斗殴/肢体冲突 | 高 | 最贴近校园冲突 |
| Assault | 单向攻击/疑似霸凌 | 高 | 可作为霸凌近似映射 |
| Abuse | 异常攻击/虐待行为 | 高 | 可并入攻击类 |
| People_falling | 跌倒/昏厥 | 高 | 贴近赛题要求 |
| Vandalism | 破坏公共设施 | 高 | 贴近赛题要求 |
| Robbery / Burglary / Stealing | 可疑闯入/财物风险 | 中高 | 可作为校园财产风险 |
| Arson / Fire / Explosion | 火情/极端危险行为 | 高 | 校园高危事件 |
| Object_falling | 高空坠物/物体坠落风险 | 中高 | 可扩展展示 |
| RoadAccidents / Traffic_accident | 校园交通风险 | 中 | 可选展示 |
| Normal | 正常行为 | 低 | 不预警 |

### 重要说明

以下类别在当前仓库中没有直接现成支持，不应在 2 小时版本里承诺：

- 吸烟
- 打电话
- 长时间使用手机
- 摄像头遮挡
- 翻越围栏
- 多人角色精确区分

若必须在方案中提及，应表述为：

> 当前版本仅预留接口，不在 2 小时 Demo 中实现。

## 7.5 解释生成层

### 功能目标

为每个预警事件生成一段“原因说明”。

### 优先级策略

1. 优先读取 DeepSeek 推理文本
2. 若无可用推理文本，则读取 LLaVA 描述文本
3. 若文本过长，则做摘要截断

### 事件与文本匹配方式

对每个事件 `[start_frame, end_frame]`：

- 找到与其重叠度最大的 coarse/fine clip
- 选取对应推理文本作为事件解释

### 输出字段建议

- `reason_text`
- `reason_source`
- `support_clip`

## 7.6 展示层

### 展示内容

页面至少展示：

1. 视频播放器
2. 异常分数曲线
3. 当前事件列表
4. 当前事件说明
5. 风险等级与标签

### 最省实现方式

优先级建议如下：

1. `Gradio` 单文件页面
2. 若环境无 `Gradio`，则退化为 `Python + 静态 HTML 报告`

### 页面最小布局

```text
左侧：视频播放
右侧：事件列表 + 事件说明
下方：异常分数曲线 + 导出按钮 + 历史记录
```

## 7.7 导出层

### 需要导出的内容

- 识别日志 `json/csv`
- 异常切片 `mp4`
- 分数曲线图 `png`
- 历史记录缓存 `jsonl`

### 日志字段建议

| 字段 | 含义 |
| --- | --- |
| event_id | 事件编号 |
| video_name | 视频名 |
| source_class | 原始类别 |
| campus_label | 校园标签 |
| risk_level | 风险等级 |
| start_sec | 起始时间 |
| end_sec | 结束时间 |
| peak_score | 峰值分数 |
| reason_text | 原因说明 |
| clip_path | 导出切片路径 |
| created_at | 生成时间 |

## 7.8 历史记录缓存

最简单实现方式：

- 每次分析后向 `outputs/history/history.jsonl` 追加一条记录

每条记录保存：

- 视频名
- 事件数量
- 高危事件数量
- 导出文件路径
- 分析时间


## 8. 模块划分建议

2 小时版本建议新增一个轻量目录，例如：

```text
campus_demo/
├── app.py
├── config.py
├── result_loader.py
├── event_builder.py
├── mapper.py
├── exporter.py
└── history.py
```

### 各模块职责

#### `config.py`

- 管理结果路径
- 管理视频路径
- 管理类别映射
- 管理阈值配置

#### `result_loader.py`

- 读取最终分数 JSON
- 读取 caption JSON
- 读取 reasoning JSON
- 统一输出结构

#### `event_builder.py`

- 平滑分数
- 阈值切段
- 事件合并
- 风险打级

#### `mapper.py`

- 原始类别到校园类别映射
- 风险等级映射
- 文本关键词补充映射

#### `exporter.py`

- 导出 CSV/JSON
- 调用 ffmpeg 或 OpenCV 导出切片
- 生成曲线图

#### `history.py`

- 保存历史记录
- 读取历史记录

#### `app.py`

- 页面入口
- 连接各模块
- 响应用户操作

## 10. 详细研发步骤

## 10.1 第 1 阶段：整理配置与样本

目标：

- 选出 3 到 5 个最适合展示的视频样本
- 确认其对应结果 JSON 路径

建议样本：

- `Normal_Videos_xxx`：正常样本
- `Fightingxxx`：打架样本
- `Vandalismxxx`：破坏设施样本
- `Robbery/Burglary/Stealingxxx`：可疑闯入样本

若后续能补到 MSAD 视频，则再加入：

- `People_falling_xxx`

产出：

- 一个固定样本配置表

## 10.2 第 2 阶段：实现结果加载器

目标：

- 输入视频名
- 输出统一结果对象

关键点：

- 处理不同 JSON 的结构差异
- 统一分数字段
- 统一文本字段
- 对齐视频名与路径

## 10.3 第 3 阶段：实现事件切分器

目标：

- 从逐帧分数中切出异常事件

关键点：

- 平滑
- 阈值
- 合并
- 最短时长过滤

## 10.4 第 4 阶段：实现校园映射器

目标：

- 输出“校园标签 + 风险等级 + 事件说明”

关键点：

- 数据集类别映射
- 风险等级配置
- 文本说明提取

## 10.5 第 5 阶段：实现导出器

目标：

- 导出日志
- 导出切片
- 导出曲线图

关键点：

- 切片优先使用 `ffmpeg`
- 若 `ffmpeg` 不可用，则降级为 OpenCV 切帧写视频

## 10.6 第 6 阶段：实现展示入口

目标：

- 做一个可演示页面

最小交互：

1. 选择视频
2. 点击分析
3. 显示事件列表
4. 播放视频
5. 导出日志和切片

## 12. 验收标准

## 12.1 功能验收

系统至少应满足：

1. 能打开一个本地样本视频。
2. 能显示该视频的异常曲线。
3. 能输出不少于 1 条异常事件。
4. 能显示校园标签与风险等级。
5. 能显示事件说明文本。
6. 能导出日志文件。
7. 能导出异常切片。
8. 能保存历史记录。

## 12.2 演示验收

建议至少准备以下 3 类样本：

- 正常视频
- 打斗视频
- 破坏/可疑行为视频

演示流程建议：

1. 先播放正常视频，说明系统不会频繁误报。
2. 再播放打斗样本，展示高风险告警。
3. 最后播放破坏或可疑闯入样本，展示日志导出和切片导出。

## 12.3 文字表述验收

在答辩或文档中应使用如下口径：

> 本系统当前版本采用“训练自由结果复用 + 业务化预警封装”的快速研发路径，优先验证校园安防预警闭环的工程可行性。当前交付重点在于事件定位、预警展示、日志导出与系统可解释性，而非重新训练专用行为识别模型。


## 13. 风险与应对

## 13.1 风险一：MSAD 对应视频不在本地

### 影响

- 无法直接展示 `People_falling` 视频回放

### 应对

- 页面演示以 UCF 本地视频为主
- 方案文档中引用 MSAD 结果与类别映射能力
- 把跌倒类作为“已有结果储备，可在补充视频后接入”

## 13.2 风险二：结果目录命名复杂

### 影响

- 手工拼路径容易出错

### 应对

- 在 `config.py` 中统一维护路径表
- 第一版只支持固定样本，不做自动全盘扫描

## 13.3 风险三：解释文本过长

### 影响

- 页面不易展示

### 应对

- 限制显示前 200 到 300 字
- 提供“展开全文”按钮或折叠区

## 13.4 风险四：导出切片依赖外部工具

### 影响

- 若环境缺少 `ffmpeg`，切片可能失败

### 应对

- 优先 `ffmpeg`
- 失败后自动降级为 OpenCV 输出

## 13.5 风险五：题目要求与 Demo 能力存在落差

### 影响

- 审阅者可能质疑“为何不是实时系统”

### 应对

明确表述：

- 当前版本是工程最小闭环
- 先完成“感知 -> 预警 -> 导出 -> 回放 -> 记录”
- 后续再把离线结果替换为在线推理接口
