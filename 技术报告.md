# A28 校园安防双视频输入系统技术报告

## 1. 项目背景

本项目面向 A28《面向校园安防的视频行为感知与异常事件智能预警》赛题，目标是在现有 `VADTree` 仓库基础上，构建一个可比赛演示的校园安防视频分析系统。

当前仓库的核心优势在于：

- 已有 `VADTree` 论文级异常检测与解释链路
- 已有 `campus_demo` 控制台与报告生成基础
- 已有事件导出、历史记录、SSE 状态流等工程外壳
- 已有可复用的数据加载、事件构建、映射与导出模块

但当前系统的主要问题也很明确：

- 更偏向“已有结果演示”，不是“真实双输入分析后端”
- 上传视频虽然能落盘，但对新视频的真实分析链路不完整
- RTSP 输入仍为预留状态，尚未真正接入实时处理

因此，本次技术方案的核心目标不是推翻现有结构，而是在 `campus_demo` 基础上完成前端、接口、后端三位一体的工程闭环。

---

## 2. 建设目标

本次系统建设面向比赛演示，要求前端、接口、后端围绕统一主线展开：

### 2.1 正式支持两类视频输入

- 普通视频上传
- RTSP 在线视频

### 2.2 统一处理规则

两类输入统一采用以下时序处理策略：

- 基础段长度固定为 `5` 帧
- 第 `1` 个基础段只缓存，不触发识别
- 从第 `2` 个基础段开始，使用“上一段 5 帧 + 当前段 5 帧”组成一个 `10` 帧分析窗口
- 每个窗口执行一次识别
- 多个连续窗口的结果聚合为事件

### 2.3 面向比赛的完整闭环

系统最终需要支持：

1. 输入视频或连接 RTSP
2. 创建分析任务
3. 实时或准实时展示状态与告警
4. 查看并编辑事件日志
5. 导出 JSON / CSV / 异常切片 / ZIP
6. 保留并回看历史任务

## 3. 现有基础与当前问题

## 3.1 前端基础

当前前端主要基于 `VADTree/campus_demo/console.html`，已经具备：

- 控制台首页
- 输入切换
- 任务状态显示
- SSE 刷新
- 日志编辑
- 导出入口
- 历史回看

它已经不是纯静态页面，而是一个可扩展的轻量控制台。

## 3.2 后端基础

当前后端入口在 `VADTree/campus_demo/app.py`，已具备：

- 上传接口
- 任务接口
- SSE 状态流
- 事件查询与编辑
- 导出
- 历史任务
- 健康检查

此外，现有模块还包括：

- `result_loader.py`：加载已有结果缓存
- `event_builder.py`：由分数生成事件
- `mapper.py`：行为与校园风险映射
- `exporter.py`：生成报告、导出切片、打包 ZIP

## 3.3 主要问题

虽然已有工程骨架，但还存在三个关键问题：

### 问题一：上传视频链路仍偏结果复用

当前上传视频后，系统更擅长“读取已有缓存结果”而不是“对这个新视频本身做真实分析”。

### 问题二：RTSP 仍是占位能力

前端和任务模型中已经预留了 RTSP，但后端并未真正实现：

- 建立连接
- 持续拉流
- 录制本地视频
- 断流重连
- 实时切段与识别

### 问题三：前后端虽有接口，但未围绕双输入方案做统一约束

尤其在以下方面还缺少统一：

- 双输入共用的任务字段
- 双输入共用的事件结构
- 双输入共用的导出模型
- 双输入共用的历史记录结构

## 4. 总体技术方案

为了兼顾比赛演示、工程复杂度和现有代码基础，本项目采用“轻量控制台 + 文件型后端 + 窗口化 VADTree 输入”的技术路线。

整体结构如下：

```text
前端控制台
  -> 统一 API 层
  -> 任务调度层
  -> 视频切段与窗口构造层
  -> 窗口识别层
  -> 事件聚合层
  -> 导出与历史层
```

这套结构的关键思想是：

- 前端不关心视频来自上传还是 RTSP，只关心任务、事件和导出
- 接口层不区分完全不同的协议体系，而是统一围绕 `job` 工作
- 后端用统一的“5 帧段 + 10 帧窗口”规则，把两类输入规整成同一种分析单元
- 事件、导出、历史三部分采用完全一致的数据模型

## 5. 前端设计方案

## 5.1 前端目标

前端的职责不是重做整个系统，而是在现有 `campus_demo` 基础上形成一个比赛演示控制台。

它必须完成的事情包括：

- 展示输入源选择
- 创建分析任务
- 显示任务状态、FPS、延迟、事件数
- 显示最近预警
- 支持事件日志查看与编辑
- 支持导出
- 支持历史任务回看

## 5.2 页面结构

前端建议保持现有控制台结构，并围绕以下五个区域组织：

1. 输入源与任务创建区
2. 任务运行态概览区
3. 预警日志与人工复核区
4. 导出区
5. 历史任务区

页面应清晰表达如下流程：

`输入 -> 分析 -> 预警 -> 复核 -> 导出 -> 历史回看`

## 5.3 前端核心交互

### 普通视频

- 选择文件
- 上传视频
- 创建任务
- 查看进度、FPS、事件
- 查看并编辑日志
- 导出结果

### RTSP

- 输入 RTSP 地址
- 创建任务
- 查看 `stream_state`
- 查看实时事件
- 停止采集
- 导出与回看

## 5.4 前端约束

- 不引入新的重量级前端框架
- 保持中文文案
- 优先满足桌面端比赛演示
- 所有按钮都必须有真实后端行为支撑

## 6. 接口设计概述

## 6.1 接口设计原则

接口层的目标不是暴露复杂内部流程，而是为前端提供统一、稳定、可轮询或可订阅的任务接口。

因此本项目采用统一的 `job` 模型：

- 上传视频和 RTSP 都通过 `POST /jobs` 创建任务
- 任务详情通过 `GET /jobs/{job_id}` 获取
- 实时状态通过 `GET /jobs/{job_id}/stream` 获取
- 当前事件通过 `GET /jobs/{job_id}/events` 获取

## 6.2 接口职责分层

### 输入接口

- 上传视频：`POST /campus_demo/api/sources/upload`
- 创建任务：`POST /campus_demo/api/jobs`

### 状态接口

- 单任务查询：`GET /campus_demo/api/jobs/{job_id}`
- SSE 状态流：`GET /campus_demo/api/jobs/{job_id}/stream`
- 健康检查：`GET /campus_demo/api/health`

### 结果接口

- 获取事件：`GET /campus_demo/api/jobs/{job_id}/events`
- 编辑事件：`PATCH /campus_demo/api/events/{event_id}`
- 导出结果：`POST /campus_demo/api/jobs/{job_id}/export`
- 获取历史：`GET /campus_demo/api/history`

## 6.3 接口对象

系统统一围绕以下对象工作：

- `AnalysisJob`
- `AlertEvent`
- `UploadedSource`

完整字段、请求和响应格式见 [接口.md](./接口.md)。

## 7. 后端设计方案

## 7.1 后端目标

后端的核心任务是把当前系统从“可展示已有结果”升级为“可真实处理双输入”的视频分析后端。

后端不是单纯新增几个接口，而是要真正完成：

- 视频接入
- 视频切段
- 窗口识别
- 事件生成
- SSE 推送
- 导出与历史闭环

## 7.2 分层结构

后端建议保持 `campus_demo` 结构不变，在其上扩展六层能力：

### 1. 输入接入层

职责：

- 接收上传文件
- 接收 RTSP 地址
- 创建统一任务对象
- 为后续链路准备输入上下文

### 2. 视频切段与窗口构造层

职责：

- 每 `5` 帧切一个基础段
- 缓存“上一段”
- 构造“上一段 + 当前段”形成的 `10` 帧窗口

这是普通视频和 RTSP 两条链路能够统一的关键。

### 3. 窗口识别层

职责：

- 接收 `10` 帧窗口
- 组织成可输入到 `VADTreeWindowRunner` 的最小识别单元
- 输出：
  - `confidence`
  - `behavior_type`
  - `reason_text`

### 4. 事件聚合层

职责：

- 将窗口结果转为事件
- 合并时间连续、类别一致的窗口
- 生成统一事件对象

### 5. 任务状态 / SSE 层

职责：

- 维护任务生命周期
- 更新进度、FPS、延迟、当前时间点、事件数
- 持续推送 SSE

### 6. 导出与历史层

职责：

- 保存当前事件版本
- 生成报告与导出物
- 写入历史记录

## 7.3 普通视频处理流程

普通视频链路的处理流程如下：

1. 视频文件上传并落盘
2. 基于 `source_id` 创建任务
3. 后端打开完整视频，读取总帧数、时长、FPS
4. 每 `5` 帧切出一个基础段
5. 第 `1` 段只缓存
6. 从第 `2` 段开始，取“上一段 + 当前段”组成 `10` 帧窗口
7. 每个窗口调用 `VADTreeWindowRunner`
8. 识别结果进入事件聚合器
9. 事件通过 SSE 推送，任务完成后生成报告和导出物

这一链路的特点是：

- 有完整文件
- 有确定总时长
- 可以计算 `progress`
- 不需要重连与录制逻辑

## 7.4 RTSP 处理流程

RTSP 链路的处理流程如下：

1. 接收 RTSP 地址并创建任务
2. 后端建立连接
3. 持续拉流与解码
4. 同步录制本地 mp4
5. 每收齐 `5` 帧形成一个基础段
6. 第 `1` 段只缓存
7. 从第 `2` 段开始形成 `10` 帧窗口
8. 每个窗口调用 `VADTreeWindowRunner`
9. 识别结果进入事件聚合器
10. 通过 SSE 实时推送状态与事件
11. 停止任务后收尾、导出并写历史

这一链路的特点是：

- 持续输入
- 无固定完成度
- 必须支持录制
- 必须支持断流重连

## 7.5 统一事件规则

窗口级识别结果不能直接展示为最终日志，必须经过事件聚合。

统一规则如下：

- `confidence < 0.50`：不生成事件
- `0.50 <= confidence < 0.75`：生成 `medium`
- `confidence >= 0.75`：生成 `high`

若相邻窗口：

- `behavior_type` 一致
- 时间连续或间隔可忽略

则自动合并为同一事件。

合并后的事件：

- `start_sec` 取最早时间
- `end_sec` 取最晚时间
- `confidence` 取峰值
- `reason_text` 取峰值窗口的说明

## 7.6 运行态字段

为了支撑双输入，后端统一补充以下任务字段：

- `segment_frames`
- `window_frames`
- `processed_segments`
- `analyzed_windows`
- `buffered_segments`
- `source_fps`
- `processing_fps`
- `stream_state`

补充说明：

- `current_fps` 为兼容前端保留，语义等于 `processing_fps`
- 普通视频 `progress_mode=determinate`
- RTSP `progress_mode=indeterminate`

## 7.7 导出与历史闭环

两类输入必须共用同一套导出和历史模型。

统一导出物包括：

- `events.current.json`
- `events.current.csv`
- `analysis.current.json`
- `clips_manifest.json`
- `report_bundle.zip`

所有切片导出都从本地视频源裁剪：

- 上传视频：从上传文件裁剪
- RTSP：从录制文件裁剪

这样可以保证：

- 可重复导出
- 可历史回看
- 不依赖在线流再次连接

## 8. 前端、接口、后端如何协同

这套系统的工程价值不在于某一个页面、某一个接口或某一个模块，而在于三者形成完整闭环。

## 8.1 协同关系

### 前端

负责：

- 输入来源选择
- 创建任务
- 展示运行态
- 展示事件与告警
- 日志编辑
- 导出与历史回看

### 接口

负责：

- 为前端提供稳定契约
- 将任务、事件、导出、历史统一到同一种数据模型
- 对外屏蔽后端内部复杂度

### 后端

负责：

- 真正读取和处理视频
- 管理任务与运行态
- 将窗口识别结果转为事件
- 生成导出与历史

## 8.2 典型链路一：普通视频

```text
前端选择视频
  -> 调用上传接口
  -> 获取 source_id
  -> 创建 upload 任务
  -> 后端读取完整视频并切段
  -> 后端组装 10 帧窗口
  -> 后端调用 VADTreeWindowRunner
  -> 后端生成事件
  -> SSE 推送给前端
  -> 前端查看/编辑日志
  -> 后端导出并写历史
```

## 8.3 典型链路二：RTSP

```text
前端输入 RTSP 地址
  -> 创建 rtsp 任务
  -> 后端连接 RTSP 并录制
  -> 每 5 帧形成基础段
  -> 组装上一段 + 当前段窗口
  -> 后端调用 VADTreeWindowRunner
  -> 事件聚合
  -> SSE 持续推送
  -> 前端查看实时事件
  -> 用户停止任务
  -> 后端收尾、导出、写历史
```

## 11. 风险与建议

## 11.1 当前主要风险

- 上传视频真实分析链路尚未完全实现
- RTSP 仍未真正接入实时运行时
- 窗口化 VADTree 需要工程适配，不可直接照搬整段视频离线链路

## 11.2 工程建议

- 不重做前端，只扩展现有 `campus_demo`
- 不改变接口风格，只扩展现有任务模型
- 不引入数据库，继续使用文件落盘
- 优先完成上传与 RTSP 两条主链路，再考虑训练/评测等次要功能

---

## 12. 结论

本项目的技术路线不是简单地“加几个接口”或“补几个页面”，而是围绕比赛演示构建一个完整的视频分析闭环系统。

这套方案的关键点在于：

1. 前端、接口、后端围绕同一套任务模型协同
2. 普通视频和 RTSP 统一为“5 帧基础段 + 10 帧窗口”的处理方式
3. 所有窗口结果统一汇聚为事件，再进入 SSE、导出和历史

只要按这套技术路线推进，当前 `VADTree` 仓库就能从“离线结果演示项目”升级为“可比赛展示的校园安防双输入分析系统”。
