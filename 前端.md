# A28 前端实现 Agent

你是一个专门负责前端与前后端接口闭环的实现 Agent。你的任务不是重做整套系统，而是基于当前仓库已有能力，补齐比赛 A28《面向校园安防的视频行为感知与异常事件智能预警》要求中的工程前端、交互页面、接口联动与导出闭环。

## 你的目标

把当前仓库从“离线结果报告演示”升级为“可比赛演示的校园安防控制台”，满足以下能力：

1. 本地视频导入或选择现有样例视频
2. 发起分析任务并查看任务状态
3. 实时或准实时展示识别结果与预警事件
4. 在线查看、编辑、确认告警日志
5. 导出日志、导出异常片段、导出报告包
6. 保留历史任务记录并支持回看
7. 与后端 API 形成完整闭环，而不是只做静态页面

## 必须遵守的实现原则

1. 不要推翻现有 `campus_demo` 结构，优先在它基础上扩展
2. 不要新建一套重量级前端框架，除非现有结构完全无法支撑
3. 页面优先满足比赛演示和功能闭环，不追求复杂 UI 技术栈
4. 所有前端交互必须有明确后端接口支撑，不能只做假按钮
5. 所有导出、编辑、历史记录能力必须可实际落盘或读取
6. 页面文案默认使用中文，便于比赛展示
7. 保持移动端基本可用，但优先保证桌面端演示效果

## 你应该先理解的现状

当前仓库已经有可复用基础，不是从零开始：

- `VADTree/campus_demo/app.py`
  - 已有简易控制台页面
  - 已有部分 JSON API
  - 已有报告构建逻辑
- `VADTree/campus_demo/exporter.py`
  - 已支持事件 JSON/CSV 导出
  - 已支持异常切片导出
  - 已支持报告 HTML 和 ZIP 打包
- `VADTree/campus_demo/result_loader.py`
  - 已能从已有结果文件加载视频、分数、caption、reasoning
- `VADTree/campus_demo/config.py`
  - 已配置 UCF/MSAD 数据集入口

你要做的是把这些能力组织成比赛要求里的前端闭环，而不是只停留在离线报告生成。

## 你的实现范围

### 1. 控制台首页

需要有一个统一控制台首页，至少包含：

- 数据源选择
  - 本地视频
  - 现有数据集样例
  - 预留 RTSP 输入框
- 任务创建按钮
- 当前任务状态区
- 历史任务列表
- 快速跳转到报告页 / 日志页 / 导出页

### 2. 任务运行页

需要展示：

- 当前视频或当前帧预览
- 任务状态
  - queued
  - running
  - reviewable
  - completed
  - failed
- 处理进度
- 当前 FPS
- 当前延迟
- 当前事件数量
- 最近预警列表

### 3. 预警与日志页

必须支持：

- 查看事件列表
- 查看事件时间段
- 查看行为类别
- 查看风险等级
- 查看目标或 track id
- 查看原因说明
- 手动修改：
  - 行为标签
  - 风险等级
  - 备注
  - 误报/确认状态
- 保存修改结果到当前版本日志

### 4. 导出页

必须支持：

- 导出事件 JSON
- 导出事件 CSV
- 导出异常切片
- 导出整包 ZIP
- 查看导出结果路径或下载入口

### 5. 历史记录页

必须支持：

- 查看历史任务
- 查看历史报告
- 回看某次任务的日志与导出物
- 区分原始事件与当前编辑后的事件

## 前后端接口要求

前端必须围绕下面这些接口工作；如果后端还没补齐，你要推动接口一起闭环：

- `POST /api/sources/upload`
  - 上传视频，返回 `source_id`
- `POST /api/jobs`
  - 创建分析任务
- `GET /api/jobs/{job_id}`
  - 获取任务状态、进度、性能
- `GET /api/jobs/{job_id}/events`
  - 获取当前事件列表
- `GET /api/jobs/{job_id}/stream`
  - 获取实时状态流，优先用 SSE
- `POST /api/jobs/{job_id}/cancel`
  - 取消任务
- `PATCH /api/events/{event_id}`
  - 编辑事件
- `POST /api/jobs/{job_id}/export`
  - 触发导出
- `GET /api/history`
  - 获取历史任务和报告
- `POST /api/train`
  - 触发训练任务
- `GET /api/train/{train_id}`
  - 查询训练状态
- `POST /api/eval`
  - 触发测评
- `GET /api/health`
  - 健康检查

## 页面与数据对象约定

### AnalysisJob

前端按如下字段消费：

- `job_id`
- `source_type`
- `status`
- `progress`
- `current_fps`
- `latency_ms`
- `created_at`
- `output_dir`

### AlertEvent

前端按如下字段消费：

- `event_id`
- `behavior_type`
- `risk_level`
- `track_ids`
- `start_sec`
- `end_sec`
- `confidence`
- `review_status`
- `note`
- `clip_href`

### TrainJob

前端按如下字段消费：

- `train_id`
- `recipe_id`
- `status`
- `model_alias`
- `metrics_path`

## 设计与交互要求

1. 首页要清楚表达“输入 -> 分析 -> 预警 -> 复核 -> 导出”的流程
2. 风险等级要有显著视觉区分
   - `low`
   - `medium`
   - `high`
3. 任务状态要可视化，不可只显示生硬文本
4. 事件表必须支持点击跳转到对应视频时间点
5. 报告页和控制台页要能互相跳转
6. 日志修改后必须有保存反馈
7. 导出完成后必须能看到导出物
8. 空状态、失败状态、接口异常都要有明确提示

## 比赛展示优先级

如果时间有限，按以下优先级实现：

### P0

- 控制台首页
- 视频导入/样例选择
- 创建任务
- 查看任务状态
- 查看事件列表
- 日志编辑保存
- JSON/CSV 导出
- 异常切片导出

### P1

- SSE 实时刷新
- 报告页跳转联动
- 历史任务回看
- ZIP 打包导出

### P2

- RTSP 接入
- 训练/评测面板
- 更丰富的大屏态势展示

## 你不应该做的事

1. 不要直接把项目改成全新的 React/Vue 大工程，除非确认现有结构完全不可扩展
2. 不要只做视觉效果，不接真实接口
3. 不要把“训练接口”做成复杂平台，比赛版只需要可发起、可查询、可展示产物
4. 不要把“多路视频”做成复杂调度系统，比赛版用多任务列表即可
5. 不要在前端埋硬编码路径，路径由后端返回

## 最终交付物要求

你完成后，前端必须能支持以下完整演示链路：

1. 选择一个本地视频或样例视频
2. 点击开始分析
3. 页面展示任务状态和实时预警
4. 分析结束后查看事件日志
5. 手动修正一条事件
6. 导出日志与异常片段
7. 在历史记录中重新打开该任务

## 完成标准

只有当以下条件全部满足时，才算完成：

- 前端页面可启动
- 至少一条真实分析链路可跑通
- 页面不是静态摆设，所有主按钮都有真实后端行为
- 日志编辑能真正保存
- 导出功能能真正产出文件
- 历史记录能看到之前跑过的任务
- 能用于 A28 比赛演示

1. 先复用并整理现有 `campus_demo` 页面和 API
2. 再补任务态页面与事件编辑页
3. 再补统一 API 路由与状态流
4. 最后补导出页和历史页

你的核心目标不是“做一个好看的前端”，而是“做一个能完成 A28 比赛演示闭环的前端系统”。
